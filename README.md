# open-finance-br-cli

CLI Python para consumir APIs padronizadas do Open Finance Brasil como
consumidor. Ele usa apenas biblioteca padrao (`urllib`, `argparse`, `json`) e
gera respostas JSON coloridas no terminal.

## Specs usadas

- Accounts v2.4.2: https://raw.githubusercontent.com/OpenBanking-Brasil/openapi/main/swagger-apis/accounts/2.4.2.yml
- Consents v3.0.1: https://raw.githubusercontent.com/OpenBanking-Brasil/openapi/main/swagger-apis/consents/3.0.1.yml
- Resources v3.0.0: https://raw.githubusercontent.com/OpenBanking-Brasil/openapi/main/swagger-apis/resources/3.0.0.yml
- Participantes do diretorio: https://data.directory.openbankingbrasil.org.br/participants

Os mocks retornam payloads com os campos obrigatorios dessas specs para permitir
desenvolvimento local quando nao ha instituicao real configurada.

## Instalacao

```bash
python3 -m pip install -e .
openfinance --version
```

O comando instalado no PATH e `openfinance`.

## Configuracao

```bash
openfinance config set \
  --base-url https://api.banco.com.br/open-banking/accounts/v2 \
  --token-url https://auth.banco.com.br/oauth2/token \
  --client-id "$CLIENT_ID" \
  --client-secret "$CLIENT_SECRET" \
  --certificate /path/cert.pem \
  --certificate-key /path/key.pem
```

URLs por familia podem ser configuradas separadamente quando a instituicao
publicar bases diferentes:

```bash
openfinance config set \
  --accounts-url https://api.banco.com.br/open-banking/accounts/v2 \
  --consents-url https://api.banco.com.br/open-banking/consents/v3 \
  --resources-url https://api.banco.com.br/open-banking/resources/v3
```

Arquivos locais:

```bash
openfinance config path
openfinance config show
```

Por padrao os arquivos ficam em `~/.openfinance/config.json` e
`~/.openfinance/token_cache.json`. Para testes, use `OPENFINANCE_CONFIG` e
`OPENFINANCE_TOKEN_CACHE`.

## Token OAuth2

```bash
openfinance token --scope "openid accounts consents resources"
openfinance token --force
```

Quando nenhuma instituicao real ou credencial OAuth esta configurada, o CLI
retorna e cacheia um token mock. Em uso real, configure `token_url`, `client_id`
e `client_secret`; o token e obtido por OAuth2 `client_credentials` com HTTP
Basic Auth e corpo `grant_type=client_credentials`.

## Contas

```bash
openfinance accounts
openfinance accounts --page 1 --page-size 25 --account-type CONTA_DEPOSITO_A_VISTA
openfinance account mock-account-001
openfinance balances mock-account-001
openfinance transactions mock-account-001 --from-booking-date 2026-01-01 --to-booking-date 2026-05-25
openfinance transactions-current mock-account-001
```

## Consentimentos

Criar consentimento com payload montado pelo CLI:

```bash
openfinance consents create \
  --cpf 12345678901 \
  --permissions ACCOUNTS_READ ACCOUNTS_BALANCES_READ ACCOUNTS_TRANSACTIONS_READ RESOURCES_READ \
  --expiration-date-time 2026-12-31T23:59:59Z
```

Usar payload JSON proprio:

```bash
openfinance consents create --payload consent.json
openfinance consents get urn:openfinancebrcli:example
openfinance consents delete urn:openfinancebrcli:example
```

## Diretorio e recursos

```bash
openfinance participants --limit 5
openfinance participants --sandbox --limit 5
openfinance resources --page 1 --page-size 25
```

`participants` consulta o endpoint publico do diretorio central, salvo quando
`--mock`, `--mock-response` ou `config set --mock true` forem usados.

## Mock

Sem `base_url`, os comandos de instituicao (`accounts`, `account`, `balances`,
`transactions`, `transactions-current`, `consents`, `resources`) retornam mocks
schema-shaped. Tambem e possivel forcar:

```bash
openfinance --mock accounts
openfinance config set --mock true
```
